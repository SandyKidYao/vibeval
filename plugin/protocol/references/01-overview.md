# vibeval Protocol — Overview

vibeval Data Protocol v0.2

vibeval tests are organized by **feature module**, each containing two types of data:

- **Dataset** — Test inputs, including synthetic data and evaluation criteria
- **Result** — Test outputs, including traces (process records) and evaluation results

## Unified Model

All tests are N-turn interactions. Single-turn tests are the special case where turns=1.
Each turn has a fixed structure: **input → steps[] → output**.

## Directory Structure

```
tests/vibeval/
├── {feature_name}/                # Organized by feature module
│   ├── contract.yaml              # Negotiated standard (requirements, quality criteria)
│   ├── analysis/                  # /vibeval analyze phase artifacts (optional)
│   │   └── ...
│   ├── design/                    # /vibeval design phase artifacts (optional)
│   │   └── ...
│   ├── datasets/                  # Synthetic test data + evaluation criteria
│   │   └── {dataset_name}/
│   │       ├── manifest.yaml
│   │       └── *.json | *.yaml
│   ├── results/                   # Test run outputs
│   │   └── {run_id}/
│   │       ├── summary.json
│   │       └── {test}__{item}.result.json
│   ├── comparisons/               # Comparison results
│   │   └── {run_a}_vs_{run_b}.json
│   └── tests/                     # Test code (generated by VibeCoding)
│       ├── conftest.py
│       └── test_*.py
```

Each feature directory is self-contained.

## Data Flow

```
VibeCoding → datasets/ + tests/
           → run tests → results/{run_id}/*.result.json
           → vibeval judge {feature} {run_id}  → evaluation
           → vibeval compare {feature} run_a run_b → comparison
```

## Agent Features

For features where the AI under test is an Agent with a tool catalogue (custom tools, MCP tools, or sub-agents), `analysis.yaml` includes a top-level `tools[]` section that serves as the shared contract between the analyze and design phases. The design phase produces a corresponding `tool_coverage[]` section in `design.yaml`. See `07-agent-tools.md` for the full protocol — inventory entry structure, static design-audit finding taxonomy, and the per-tool coverage matrix.

For non-Agent features, `tools[]` is omitted and the analyze/design flow is unchanged.
