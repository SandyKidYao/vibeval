# vibeval Protocol — Overview

vibeval Data Protocol v0.2

vibeval tests are organized by **feature module**, each containing two types of data:

- **Dataset** — Test inputs, including synthetic data and evaluation criteria
- **Result** — Test outputs, including traces (process records) and evaluation results

## Unified Model

All tests are N-turn interactions. Single-turn tests are the special case where turns=1.
Each turn has a fixed structure: **input → steps[] → output**.

## Directory Structure

```
tests/vibeval/
├── {feature_name}/                # Organized by feature module
│   ├── contract.yaml              # Negotiated standard (requirements, quality criteria)
│   ├── analysis/                  # /vibeval analyze phase artifacts (optional)
│   │   └── ...
│   ├── design/                    # /vibeval design phase artifacts (optional)
│   │   └── ...
│   ├── datasets/                  # Synthetic test data + evaluation criteria
│   │   └── {dataset_name}/
│   │       ├── manifest.yaml
│   │       └── *.json | *.yaml
│   ├── results/                   # Test run outputs
│   │   └── {run_id}/
│   │       ├── summary.json
│   │       └── {test}__{item}.result.json
│   ├── comparisons/               # Comparison results
│   │   └── {run_a}_vs_{run_b}.json
│   └── tests/                     # Test code (generated by VibeCoding)
│       ├── conftest.py
│       └── test_*.py
```

Each feature directory is self-contained.

## Data Flow

```
VibeCoding → datasets/ + tests/
           → run tests → results/{run_id}/*.result.json
           → vibeval judge {feature} {run_id}  → evaluation
           → vibeval compare {feature} run_a run_b → comparison
```

## Agent Features

`analysis.yaml` carries a required `project.execution_mode` field with two allowed values:

- `"agent"` — the project registers tools (custom function tools, MCP tools, or sub-agents) for LLM consumption. `analysis.yaml` MUST include a top-level `tools[]` inventory and `design.yaml` MUST include a corresponding `tool_coverage[]` cross-reference. The Agent-features flow in `07-agent-tools.md` applies.
- `"non_agent"` — no tool catalogue is exposed. `tools[]` and `tool_coverage[]` are omitted, and the analyze/design flow runs without the Agent-features dimensions.

The analyze skill determines the value once, by scanning the source for tool registration sites; every downstream consumer (design, evaluator, consultant) reads `project.execution_mode` instead of re-scanning.

See `07-agent-tools.md` for the full Agent-features protocol: tool inventory entry structure, static design-audit finding taxonomy, per-tool coverage matrix, Allowed Spec Patterns Per Dimension, and the strengthened `tool_coverage[]` invariant.
